⚠️ Problem Statement

Small clinics and hospitals rely on manual processes and face these challenges:

Appointment clashes: Manual booking leads to double-booking, same slot given to multiple patients, and no visibility of doctor availability.
Paper-based prescriptions: Prescriptions get lost, damaged, or illegible; no digital backup or searchable history.
Inefficient scheduling: Doctors and staff spend time on phone calls and registers instead of focusing on patients.
No central medical history: Past visits, prescriptions, and diagnoses are not in one place — poor continuity of care.
Poor patient experience: Long wait times, no online booking, and no way to check appointment status from home.

Goal: Build a secure, scalable digital system for appointments, prescriptions, and patient medical history.

2

💡 Our Solution

A full-stack web application with three role-based dashboards and separate permissions:

Patient

Register, view doctors by specialization, book appointments, view status, prescriptions & full medical history.

Doctor

View assigned appointments, approve/reject requests, create digital prescriptions, view patient history & daily schedule.

Admin

Add/remove doctors, view all patients & appointments, system overview dashboard and monitoring.

Each role has its own login, dashboard, and access control — no cross-access between roles.

3

🧑‍🤝‍🧑 Patient Module — Detailed

Everything a patient can do after registration and login:

Registration & Login

  • Register with name, email, password (min 6 characters); role is set as Patient.
  • Login with email and password; receive JWT token for subsequent API calls.

Doctors & Appointments

  • View list of all doctors with their name, email, and specialization.
  • Book an appointment by selecting doctor, date, and time (no double-booking — slot validation on backend).
  • View appointment status: Pending, Approved, or Rejected.

Prescriptions & History

  • View and download prescriptions (medicines, dosage, duration, remarks).
  • View complete medical history: all past appointments and prescriptions in one place.
4

🩺 Doctor Module — Detailed

Features available only to users with the Doctor role:

Login & Dashboard

  • Secure login; dashboard shows daily schedule and counts (today’s appointments, pending requests, prescriptions created).

Appointments

  • View all appointments assigned to the doctor (patient name, date, time, status).
  • Approve or reject pending appointment requests with one click.

Prescriptions

  • Create digital prescriptions only for approved appointments: add medicines with name, dosage, duration, and optional remarks.
  • View list of all prescriptions created by the doctor.

Patient History

  • Select a patient (from own appointments) to view their full medical history: past appointments and prescriptions.
5

🛡️ Admin Module — Detailed

Features available only to users with the Admin role:

Login & Overview

  • Login with admin credentials (e.g. created via seed script).
  • System overview dashboard: total doctors, patients, appointments, prescriptions (counts).

Doctor Management

  • Add new doctors: name, email, password, specialization.
  • View list of all doctors; remove a doctor from the system.

Monitoring

  • View all patients (name, email, join date).
  • View all appointments across the system (patient, doctor, date, time, status).

Admin does not create prescriptions or approve appointments — that is the doctor’s role.

⚙️ Tech Stack

Frontend React 18 (functional components, hooks), React Router, Axios, Tailwind CSS, Vite
Backend Node.js, Express.js (REST API)
Database MongoDB with Mongoose ODM
Auth JWT (JSON Web Token), role-based middleware, bcrypt for password hashing
API RESTful design, express-validator for input validation, centralized error handling
UI Responsive, clean, minimal; forms with client-side validation

Industry-standard stack suitable for final year submission and scalable for future extensions.

7

🗄️ Database Models (MongoDB / Mongoose)

Three main collections with clear relationships:

User

name, email, password (hashed), role (admin | doctor | patient), specialization (for doctor only)

Appointment

patientId (ref User), doctorId (ref User), date, time, status (pending | approved | rejected)

Prescription

appointmentId (ref Appointment), medicines (array: name, dosage, duration), remarks, createdAt

Indexes and validation ensure no double-booking (same doctor, date, time) and referential integrity.

🔐 API & Security

Authentication APIs: Register (patient), Login (all roles), Get current user (protected).

Appointment APIs: List doctors, List appointments (role-filtered), Book appointment (patient), Update status approve/reject (doctor).

Prescription APIs: List prescriptions (role-filtered), Create prescription (doctor), Get single prescription.

Admin APIs: Overview counts, List patients, List all appointments, Add/List/Remove doctors.

Security: JWT in Authorization header; role-based middleware (authorize('admin'), authorize('doctor'), authorize('patient')); bcrypt for passwords; express-validator on all inputs; centralized error handler and clean API responses.

🔁 Application Flow

End-to-end flow: Login → Role-based redirect → Dashboard actions:

Patient: Book appointment (select doctor, date, time) → View status → View prescriptions & full medical history.
Doctor: See pending appointments → Approve/Reject → For approved, create prescription (medicines, dosage, remarks) → View patient history.
Admin: Add doctors (name, email, password, specialization) → View all patients & appointments → System overview.

Frontend: React with Axios; JWT sent in header; protected routes redirect unauthenticated users to login.

🧩 Non-Functional Requirements

  • Security: JWT authentication, role-based access control, bcrypt password hashing, validation on every request.
  • Performance: Lightweight REST APIs, indexed queries on appointments (doctor, date, time) to avoid slow lookups.
  • Reliability: Centralized error handling, consistent JSON responses, clear HTTP status codes.
  • Usability: Clean, responsive UI (Tailwind CSS), simple navigation, clear status badges and labels.
  • Scalability: Modular backend (routes, controllers, models, middleware) and React frontend ready for new modules.
  • Maintainability: Industry-standard folder structure, separation of concerns, reusable components and hooks.
12

🧱 System Architecture

Frontend (Client)

  • React single-page application with React Router.
  • Auth context stores logged-in user and JWT token.
  • Axios client sends API requests with Authorization header.
  • Role-based routes: /patient, /doctor, /admin.

Backend (Server)

  • Express.js REST API: /api/auth, /api/appointments, /api/prescriptions, /api/admin.
  • Middleware chain: CORS → JSON parser → protect (JWT) → authorize (role).
  • Mongoose models for User, Appointment, Prescription connected to MongoDB.
  • Error handling middleware for logging and safe responses.

The architecture separates concerns clearly and makes it easy to add new modules like billing or lab reports.
